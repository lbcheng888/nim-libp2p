# LSMR/Fabric 剩余阻塞

## 当前结论

`LSMR` 证书收敛已经不是主阻塞。

现在真正没收口的是：

**Fabric 长链在高负载下，submit ACK 后半程仍然过慢。**

## 已经排除的方向

| 方向 | 当前结论 |
|---|---|
| overlay 没起来 | 已排除，`4/4/4/4` 可以稳定达到 |
| submit 连接根本建不起来 | 已排除，`submit session ready` 已经建立 |
| ACK 还在等远端业务处理 | 已排除，ACK 已改成 inbox 接收确认 |
| `event / attestation / cert` 还共用一条串行 runner | 已排除，已经拆成三条 lane |

## 当前第一现场

当前最窄的第一现场是：

| 项目 | 当前状态 |
|---|---|
| 每个 `peer|lane` 的 submit 子流数 | 仍然只有 1 条 |
| 每条 lane 的发送方式 | 有界 pipeline，但还是单子流顺序消费 |
| 结果 | 一个慢 ACK 仍然能把整条 lane 挂住 |

所以现在的现象不是“完全不推进”，而是：

| 现象 | 含义 |
|---|---|
| `61 tx` 在 20~45 秒内能从全 0 推进到部分节点 `2~12 event`、`1~5 cert` | 主路径已经通了 |
| 但长链后半段还会被 `slow-ack` 卡住 | 并发度还不够 |

## 为什么这还是理论问题，不是调参问题

| 错误想法 | 为什么错 |
|---|---|
| 调大 ACK 超时就行 | 这只会把问题藏起来，不会提高吞吐 |
| 再继续堆 warm/retry | 单子流 lane 的结构上限没变 |
| 再把 batch 调小 | 只能换一种慢法，不能消掉慢 ACK 阻塞 |

当前剩余问题是结构性的：

**一条 lane 只有一条 submit 子流，就意味着一个慢 ACK 会继续阻塞整个 lane。**

## 下一步的正确算法

下一步不该回头改证书收敛，也不该继续猜连接。

应该直接做：

| 方向 | 规则 |
|---|---|
| 每条 lane 固定大小多子流池 | 例如每条 lane 固定 `N` 条 submit 子流 |
| 每条子流独立 inflight 窗口 | 一个慢 ACK 只影响当前子流，不影响整条 lane |
| lane 级调度只做 work stealing | 有空闲子流就拿新 batch，没有就等 |
| 账本语义不变 | `DeliveryLedger` 仍然只按一跳 ACK 推进 |

## 不该再动的地方

| 部分 | 原因 |
|---|---|
| `LSMR` 证书闭包 promote | 这部分已经收口 |
| ACK 只确认 inbox 接收 | 这条语义已经正确，不能回退 |
| cert 后清 event/att delivery | 这条状态切换也已经正确 |

## 验收口

下一轮改完后，验收不看口头解释，只看这几条：

| 项目 | 通过条件 |
|---|---|
| overlay | `10s` 内 `4/4/4/4` |
| submit 慢连接 | 不再成为主故障 |
| submit 慢 ACK | 高负载下不再持续秒级堆积 |
| `61 tx` | 4 节点都达到 `61 event / 61 cert` |
| observer 查询 | 能查到 61 条内容 |
